# Sprint NNN Retrospective — Template

*Per the AI/Human Scrum operating model adopted at Sprint 002 close.*

## Required sections

1. **Header** — sprint, period, status, prepared-by, amendments, participants
2. **Executive Summary**
3. **Sprint Goal & Result**
4. **Major Deliverables** — each research deliverable carries an explicit **Confidence Level** (High / Medium / Low) tied to evidence breadth
5. **What Went Well**
6. **What Did Not Go Well** — items with data/trust impact get RISK-IDs in docs/RISKS.md
7. **Persona Review**
8. **Reality Review** — three buckets, strictly separated: *Assumptions Challenged*, *Assumptions That Survived* (evidence-backed only), *Assumptions Not Yet Tested* (mechanism-exists ≠ value-validated)
9. **Decisions Made** (D-NNN)
10. **Open Questions** (Q-NNN)
11. **Sprint Assessment** (grades)
12. **Post-Retro Challenges** — standing section, appended after publication:

| Challenge | Raised By | Accepted? | Action |
|---|---|---|---|
| | | | |

*Purpose: capture how the team's understanding evolves after initial retrospective publication.*

13. **Most Important Lesson**
14. **Recommendation for next sprint**

## Planning rule (effective Sprint 003, per D-004)

Every user-facing story must document, before entering implementation:

- **Parent Benefit** — what the parent gains
- **Child Benefit** — what the child gains (non-punitive framing)
- **Family Benefit** — what the family gains together

No user-facing feature enters implementation without all three perspectives documented.
